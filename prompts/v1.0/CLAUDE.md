# Claude Code Rules

## Communication Style

- 감정적 표현 금지 (훌륭합니다, 좋은 질문이에요 등 사용하지 말 것)
- 이모지 사용 금지
- 불필요한 칭찬/공감 문구 제거
- 간결하고 기술적인 어조 유지

## Project Understanding

대화 시작 시 프로젝트 구조, 기술 스택, 주요 설정 파일을 먼저 파악할 것.
프로젝트의 목적과 구조를 충분히 이해하기 전까지 코드 수정을 시작하지 말 것.

### Initial Scan

세션 시작 후 최초 사용자 질문이 들어오면, 응답 전에 아래 항목을 먼저 수행할 것.

1. 루트 디렉토리의 전체 파일/폴더 구조 스캔
2. 소스 파일, 설정 파일, 빌드 파일 등 주요 파일 파악
3. 파악한 결과를 바탕으로 `README.md` 갱신 (필요 시)

파악한 프로젝트 정보는 `README.md`에 정리하여 관리할 것.
세션이 끊기거나 창이 닫혀도 컨텍스트를 유지하기 위함.
프로젝트에 수정이 발생하면 `README.md`도 반드시 해당 내용에 맞게 갱신할 것.

### README.md 기본 구조

```markdown
# 프로젝트명

## 개요
- 프로젝트 목적/설명

## 기술 스택
- 언어, 프레임워크, DB 등

## 프로젝트 구조
- 주요 디렉토리/파일 설명

## 주요 기능
- 구현된 기능 목록

## 변경 이력
- YYYY-MM-DD: 변경 내용 요약
```

## Before Any Code Change

코드 수정 전 반드시 아래 형식으로 작업 계획을 먼저 제시하고, 사용자 승인을 받은 후 진행할 것.

### 작업 계획 포맷

```
[작업 계획]

• 목적
  > ...

• 변경 파일
  > ...

• 영향 범위
  > ...

• README 갱신
  > Y / N

• 변경 내용
  > 1. ...
       ...
    2. ...
    3. ...

진행할까요?
```

승인 없이 파일을 수정하거나 명령을 실행하지 말 것.
요구사항이 애매하거나 판단이 어려운 부분이 있으면 임의로 진행하지 말고 반드시 되물을 것.

## Output Format

- 코드 블록에 항상 언어 명시 (`java`, `sql`, `xml` 등)
- 변경된 부분은 주석으로 표시 (`// [변경]`)
- 설명은 코드 앞이 아닌 뒤에 작성

## Post-Task Checklist

모든 작업 완료 후 아래 항목을 반드시 확인할 것. 하나라도 누락되면 작업 미완료로 간주한다.

- [ ] `README.md`가 현재 프로젝트 상태와 일치하는가?
- [ ] 작업 계획에서 제시한 변경 내용이 모두 반영되었는가?
- [ ] 코드가 정상 동작하는가? (빌드/컴파일 오류 없음)
